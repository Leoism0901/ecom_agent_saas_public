- **安全红线**：
  - 在执行诸如 `rm`, `git reset`, `kill` 等具有破坏性或可能导致代码丢失的终端命令前，**必须**显式向用户确认。
  - 严禁在未经用户允许的情况下，在后台悄悄启动长期运行的守护进程。

## 💻5.AI Agent 编码核心规范：数据脱敏与配置外置

在本项目中进行任何代码生成、修改或重构时，必须严格遵守以下安全与配置规范：

1. **绝对禁止硬编码敏感数据**：
   - 任何 LLM API Key、数据库密码、Redis 连接、向量库账号、JWT 密钥、外部服务 Token 等，**严禁**直接写在 `.py` 或任何代码文件中。
   - 所有环境相关的 URL、IP、端口号必须外置。

2. **环境变量优先原则**：
   - 只要遇到配置项或敏感数据，必须通过 `os.getenv("VARIABLE_NAME")` 的方式读取。
   - 确保使用 `from dotenv import load_dotenv` 加载环境变量。

3. **同步更新配置模板**：
   - 如果你在代码中引入了新的环境变量，必须立刻将其以空值或占位符的形式追加到项目根目录的 `.env.example` 文件中。
   - **绝对禁止**将真实的密钥写入 `.env.example`。

4. **安全审查**：
   - 在输出代码之前，请自行审查是否包含了敏感信息的明文。如果有，立刻替换为环境变量读取方式。


## 🏗️ 6. 项目架构红线 (Architecture)

### 数据库
- **逻辑外键、物理不建外键**：电商高并发场景下，物理 FOREIGN KEY 会引发锁表争用。`tenant_id` 等关联字段只用 `Integer + index=True`，禁止使用 `ForeignKey()`。ORM 关联通过 `primaryjoin` + `foreign_keys` 显式声明。此规则同时适用于 `app/models.py` 和 `init.sql`。
- **中文编码**：所有 SQL 文件顶部必须有 `SET NAMES utf8mb4;`，否则 mysql 客户端用 latin1 读取导致乱码。

### 依赖注入
- Router 层通过 `get_db()` 获取会话，严禁直接操作引擎
- 数据持久化逻辑必须下沉至 Service 层

### 扩展性
- 所有表预留 `metadata_json` JSON 列，新字段优先写 JSON 而非 ALTER TABLE
